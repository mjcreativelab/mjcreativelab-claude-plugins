# Smart Commit

現在の git 差分（staged + unstaged + untracked）を分析し、作業内容ごとにまとまった単位でコミットを作成する。

## オプション

| オプション        | 説明                                                   | 例                                            |
| ----------------- | ------------------------------------------------------ | --------------------------------------------- |
| `-p <プロンプト>` | コミット対象の選別やコミットメッセージに関する追加指示 | `/smart-commit -p chore フォルダの差分は省く` |

`-p` で渡されたプロンプトは、手順 2（コミット単位の分割）で差分のフィルタリング・分割の判断に使用する。
例:

- `/smart-commit -p e2e の変更だけコミットして` → e2e/ 配下の変更のみ対象
- `/smart-commit -p docs は後回しにして` → docs/ 配下を除外
- `/smart-commit -p WIP でまとめてコミット` → 分割せず 1 コミットにまとめる

## 手順

### 0. 早期終了チェック

変更がない場合は即終了する:

```bash
if [ -z "$(git status --porcelain)" ]; then
  echo "✅ コミットする変更がありません"
  exit 0
fi
```

変更がなければ以降のステップをすべてスキップする。

### 1. ブランチ確認・切り替え

コミット前に、現在のブランチが作業に適切かを確認し、必要に応じてブランチを切り替えまたは作成する。

#### 1-1. 状態取得

以下を並列実行:

```bash
git rev-parse --abbrev-ref HEAD   # 現在のブランチ名
git branch --list                  # ローカルブランチ一覧
git status --short                 # 変更ファイル一覧（ブランチ判定の材料）
```

#### 1-2. 判定ロジック

変更内容を分析し、以下の基準で現在のブランチの適切性を判定する:

- **`main`（または `master`）にいる場合** → 不適切。作業ブランチへの移動が必要
- **作業ブランチにいて、変更内容がそのブランチの目的と一致する場合** → 適切。そのまま手順 1 へ進む
- **作業ブランチにいるが、変更内容がそのブランチの目的と異なる場合** → 不適切。別の作業ブランチへの移動が必要

#### 1-3. 不適切な場合の処理

1. 既存のローカルブランチ一覧から、変更内容に合致するブランチがあるか確認
2. **合致するブランチがある場合**: `git stash` → `git checkout <branch>` → `git stash pop` で切り替え
3. **合致するブランチがない場合**: 変更内容からブランチ命名規則に沿ったブランチ名を生成（規則は CLAUDE.md や rules を参照。なければ `feature/<短い説明>` 形式）

#### 1-4. ユーザー確認

ブランチ切り替え・作成前に必ずユーザーに確認を取る。提案内容:

- 現在のブランチ名
- 切り替え先 or 新規作成するブランチ名
- 理由（なぜ現在のブランチが不適切か）

ブランチが適切な場合は何も表示せず、次のステップに進む。

### 2. 差分の収集

以下を並列実行して全変更を把握する:

```bash
git status
git diff
git diff --cached
git log --oneline -5
```

### 3. ブランチ連動フィルタリング + コミット単位の分割

#### 3-1. ブランチ連動フィルタリング（main 以外のブランチの場合）

現在のブランチが `main`（または `master`）**以外** の場合、ブランチ名からそのブランチの目的を推定し、**目的に合致する差分のみ** をコミット対象とする。

判定手順:

1. ブランチ名を解析する（例: `feature/issue-42-add-auth` → Issue #42 の認証機能追加に関連する変更）
2. ブランチ名に Issue 番号が含まれる場合、GitHub MCP の `issue_read` ツール（`mcp__plugin_github_github__issue_read`）でタイトル・本文を確認し、ブランチの目的をより正確に把握する
3. 全変更ファイルを「ブランチ目的に合致する変更」と「合致しない変更」に分類する
4. **合致しない変更がある場合**: コミット計画（手順 4）でどの変更を除外するかを明示し、ユーザーに確認する

除外された変更は working tree に残る（stash しない）。ユーザーが別ブランチで後からコミットできる。

> **注意**: `main` ブランチにいる場合はフィルタリングせず、全変更を対象とする（手順 0.5 でブランチ切り替えが提案されるため）。

#### 3-2. `-p` オプションによる追加フィルタリング

`-p` オプションが指定されている場合、そのプロンプトの指示を **ブランチ連動フィルタリングより優先** して適用する。

- `-p` の指示がブランチ目的と矛盾する場合は `-p` を優先する
- 例: `feature/e2e-*` ブランチで `/smart-commit -p docs の変更もコミットして` → docs/ も含める

#### 3-3. コミット単位の分割

フィルタリング後の変更を「ロールバック時に問題ない単位」で分割する。分割基準:

- **機能単位**: 1 つの機能 / 修正 / リファクタに関わるファイル群をまとめる
- **レイヤー横断 OK**: 同じ機能なら frontend + backend + test を 1 コミットにまとめる
- **設定・ドキュメント**: コードと無関係な設定変更やドキュメントは別コミットにする
- **大量の同種変更**: データ追加/削除など同種の変更は 1 コミットにまとめる

### 4. ignore すべきファイルの確認

以下に該当するファイルが untracked にある場合、コミット前にユーザーに確認する:

- `.env`、credentials、秘密鍵など機密性のあるファイル
- ビルド成果物（`dist/`、`node_modules/`、`__pycache__/`、`.pyc`）
- OS 生成ファイル（`.DS_Store`、`Thumbs.db`）
- 大きなバイナリファイル（画像・動画で明らかに一時的なもの）

該当ファイルがあれば `.gitignore` への追加を提案する。

### 5. コミット計画の提示と確認

コミットを実行する前に、以下の形式でコミット計画をユーザーに提示し、承認を得る:

```
コミット 1: ✨ feat(auth): ログイン機能を追加
  - src/auth/login.ts
  - tests/auth.test.ts

コミット 2: 📝 docs: README を更新
  - README.md
```

ブランチ連動フィルタリングで除外した変更がある場合は、除外ファイル一覧も併せて表示する:

```
⏭️ 除外（ブランチ目的外）:
  - chores/snap-collection/output/snap-registry.json
  - docs/unrelated-doc.md
```

ユーザーの承認後に実行に進む。修正指示があれば計画を調整する。

### 6. コミット実行

分割した単位ごとに順番に実行:

1. 対象ファイルを `git add`（ファイル名を明示指定、`git add .` は使わない）
2. コミットメッセージを作成して `git commit`
3. 次の単位へ

コミットメッセージの形式:

```
<emoji> <type>(<scope>): <subject>

<body>

<footer>
```

#### type（必須）

- **feat**: 新機能追加
- **fix**: バグ修正
- **docs**: ドキュメントのみの変更
- **style**: コードの意味に影響しない変更（空白、フォーマット等）
- **refactor**: バグ修正でも機能追加でもないコード変更
- **test**: テストの追加・修正
- **chore**: ビルドプロセスや補助ツールの変更
- **ci**: CI 設定の変更

#### scope（オプション）

変更の範囲を示す（英語）。リポジトリの CLAUDE.md や rules にスコープ定義があればそれに従う。なければディレクトリ構造やモジュール名から適切なスコープを判断する。

#### subject（必須）

- 日本語で「何をしたか」を簡潔に書く
- 50 文字以下
- 末尾にピリオドを付けない

#### GitMoji（必須）

コミットメッセージの先頭に GitMoji 絵文字を付ける:

| Emoji | type             | 説明                     |
| ----- | ---------------- | ------------------------ |
| ✨    | feat             | 新機能追加               |
| 🐛    | fix              | バグ修正                 |
| 📝    | docs             | ドキュメント更新         |
| 💄    | feat(ui) / style | UI・スタイル変更         |
| ♻️    | refactor         | リファクタリング         |
| ✅    | test             | テスト追加・修正         |
| 🔧    | chore            | 設定ファイル・ツール     |
| 🔥    | remove           | ファイル・コード削除     |
| 🚀    | perf             | パフォーマンス改善       |
| 🔒    | security         | セキュリティ修正         |
| 🎨    | improve          | コード構造改善           |
| ⚡    | chore            | パッケージ更新・依存関係 |
| 🌐    | feat(i18n)       | 国際化対応追加           |

#### その他ルール

- 必要に応じて本文（body）に詳細を追記
- HEREDOC 形式でコミットメッセージを渡す
- リポジトリの CLAUDE.md や rules にコミットメッセージの追加規則があればそれに従う

#### 例

```
✨ feat: ログイン機能を追加
🐛 fix(api): セッション期限切れ時の 500 エラーを修正
🔧 chore: 依存パッケージを更新
📝 docs: セットアップ手順を追記
♻️ refactor(api): ミドルウェアを統合
```

詳細例:

```
✨ feat(auth): OAuth2.0 認証フローを実装

Google と GitHub の OAuth プロバイダーに対応。
自動トークンリフレッシュとセッション管理を追加。
```

### 7. 結果報告

全コミット完了後、作成したコミット一覧を表示する:

```bash
git log --oneline -<作成したコミット数>
```

## 注意事項

- `Co-Authored-By` トレーラーは付けない
- `--no-verify` は使わない（pre-commit hook が失敗したら原因を修正して再コミット）
- リモートへの push はしない（ユーザーが明示的に指示した場合のみ）
