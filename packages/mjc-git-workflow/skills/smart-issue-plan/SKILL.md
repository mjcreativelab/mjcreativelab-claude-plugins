# Smart Issue Plan

GitHub Issue を分析し、コードベースを探索して実装計画を作成する。既存の計画がある場合は更新する。

## オプション

`-p <プロンプト>`: 計画の観点・制約に関する追加指示。

例: `-p パフォーマンスを重視して` / `-p 最小限の変更で` / `-p テスト方針も含めて`

## ツール選択

GitHub API 操作には **GitHub MCP ツール**を優先。コードベース探索には Glob, Grep, Read を使用。

## 手順

### 1. Issue の読み取り

引数から Issue 番号を抽出（`#` 除去）し、`mcp__plugin_github_github__issue_read` でタイトル・本文・ラベル・コメントを取得。存在しない or クローズ済みなら通知して終了。

### 2. 既存計画の有無を確認

Issue のコメントに「## 実装計画」を含むコメントがあるか確認。また、タイトルに「[実装計画]」を含む関連 Issue がないか `search_issues` で検索。

- **既存計画あり** → 更新モードへ（手順 6）
- **既存計画なし** → 新規作成モードへ（手順 3）

### 3. コードベース探索

Issue の内容から関連キーワード・ファイルパス・機能領域を特定し、コードベースを探索:

- 関連ファイルの特定（Glob, Grep）
- 既存の実装パターン・アーキテクチャの把握（Read）
- 依存関係・影響範囲の分析

### 4. 実装計画の作成

以下の構成で計画を作成:

```markdown
## 実装計画

### 概要
<!-- なぜこの変更が必要か、何を達成するか -->

### 影響範囲
<!-- 変更が必要なファイル・モジュールの一覧 -->

### 実装手順
<!-- 番号付きの具体的な手順 -->
1. ...
2. ...

### リスク・注意点
<!-- 既存機能への影響、互換性、パフォーマンス懸念など -->

### テスト方針
<!-- どのようにテスト・検証するか -->
```

`-p` の指示があればそれを計画の観点に反映する。

### 5. 出力先の確認と投稿

ユーザーに出力先を確認:

- **Issue コメントとして投稿**（デフォルト）→ `mcp__plugin_github_github__add_issue_comment` で投稿
- **新規 Issue として作成** → `mcp__plugin_github_github__issue_write` で「<元タイトル>[実装計画]」として作成し、元 Issue を参照リンクで紐づけ

投稿後、URL を表示して終了。

---

### 6. 更新モード

既存の計画コメント/Issue を読み取り、現在のコードベース状態と照合:

1. 前回の計画からの変更点を特定（コード変更、要件変更）
2. 計画の各セクションを再評価
3. 更新内容のプレビューをユーザーに提示
4. 承認後、既存コメントを更新（`add_issue_comment` で追記）または計画 Issue を更新（`issue_write`）

## 注意事項

- 計画は日本語で記述する
- コードベース探索は十分に行い、推測ではなく実際のコード構造に基づいた計画を立てる
- 実装の詳細（具体的なコード）は書かない — 方針と手順に留める
- Issue の要件が不明確な場合はユーザーに確認してから計画を作成する
