# Smart Issue Resolve

GitHub Issue を起点にした作業開始フローを自動化する。
Issue の内容を読み取り、適切なブランチを作成してチェックアウトし、作業に取り掛かる。

## 使い方

```
/smart-issue-resolve #134
/smart-issue-resolve #134 -p テストも書いて
```

## オプション

| オプション        | 説明                                           | 例                                        |
| ----------------- | ---------------------------------------------- | ----------------------------------------- |
| `-p <プロンプト>` | 作業に関する追加指示（実装方針・制約など）      | `/smart-issue-resolve #42 -p テストも書いて`      |

`-p` で渡されたプロンプトは、手順 5（作業の実行）で実装方針の追加指示として使用する。

## ツール選択

| 操作                         | 使用ツール                                            |
| ---------------------------- | ----------------------------------------------------- |
| Issue 読み取り               | `mcp__plugin_github_github__issue_read`                |
| git 操作（branch, checkout） | Bash                                                   |

## 前提: リポジトリ情報の取得

すべてのステップで使用する `<owner>` と `<repo>` は、以下のコマンドで動的に取得する:

```bash
git remote get-url origin | sed -E 's#.+[:/]([^/]+)/([^/.]+)(\.git)?$#\1 \2#'
```

## 手順

### 1. Issue の読み取り

引数から Issue 番号を抽出し（`#` は除去）、Issue の内容を取得する。

**MCP で実行:**

- `mcp__plugin_github_github__issue_read` (method: "get", owner: `<owner>`, repo: `<repo>`, issue_number: <Issue番号>)

取得する情報:

- タイトル
- 本文（説明・要件・受け入れ基準など）
- ラベル
- アサイン状況

Issue が存在しない、またはクローズ済みの場合はユーザーに通知して終了する。

### 2. 作業ツリーの状態確認

以下を並列実行:

```bash
git rev-parse --abbrev-ref HEAD       # 現在のブランチ名
git status --short                     # 未コミットの変更
git branch --list                      # ローカルブランチ一覧
```

- **未コミットの変更がある場合**: ユーザーに通知し、続行するか確認する。続行する場合は `git stash` してから進める
- **現在 main にいない場合**: 別のブランチで作業中の可能性があるため、ユーザーに確認する

### 3. ブランチの作成

Issue の内容からブランチ名を生成する。

#### 3-1. ブランチ命名規則

リポジトリの CLAUDE.md や rules にブランチ命名規則があればそれに従う。なければ以下の形式を使用する:

| Issue ラベル / 内容       | ブランチ形式                         | 例                              |
| ------------------------- | ------------------------------------ | ------------------------------- |
| `bug` ラベルまたはバグ修正 | `fix/<issue番号>-<短い説明>`         | `fix/134-login-error`           |
| `feature` ラベルまたは新機能 | `feat/<issue番号>-<短い説明>`      | `feat/42-add-oauth`             |
| `documentation` ラベル     | `docs/<issue番号>-<短い説明>`        | `docs/55-update-api-docs`       |
| `enhancement` ラベル       | `feat/<issue番号>-<短い説明>`        | `feat/78-improve-search`        |
| その他・判別不能           | `chore/<issue番号>-<短い説明>`       | `chore/99-update-deps`          |

- `<短い説明>` は英語 kebab-case、3〜5 単語以内
- Issue タイトルから要点を抽出して命名する

#### 3-2. 既存ブランチの確認

同じ Issue に紐づくブランチが既に存在するか確認する:

```bash
git branch --list "*<issue番号>*"
git branch -r --list "*<issue番号>*"
```

- **既存ブランチがある場合**: そのブランチをチェックアウトするか、新規作成するかユーザーに確認する
- **既存ブランチがない場合**: 新規作成に進む

#### 3-3. ユーザーに確認

ブランチを作成・チェックアウトする前に、以下を表示してユーザーの承認を得る:

- Issue タイトル・番号
- 作成するブランチ名
- ベースブランチ（通常は main）

### 4. ブランチの作成とチェックアウト

ユーザーの承認後、ブランチを作成してチェックアウトする:

```bash
git checkout main                      # main を最新にする
git pull origin main                   # リモートから最新を取得
git checkout -b <ブランチ名>           # 新規ブランチを作成してチェックアウト
```

既存ブランチをチェックアウトする場合:

```bash
git checkout <ブランチ名>
git pull origin <ブランチ名>           # リモートがあれば最新を取得
```

手順 2 で stash した変更がある場合は `git stash pop` で復元する。

### 5. 作業の実行

Issue の内容を理解した上で、実装作業を行う。

#### 5-1. 作業方針の整理

Issue の本文から以下を把握し、作業方針を整理する:

- **要件**: 何を実現すべきか
- **受け入れ基準**: どうなったら完了か（記載がある場合）
- **制約・注意点**: 特記事項（記載がある場合）

`-p` オプションが指定されている場合、その内容も作業方針に加える。

#### 5-2. 実装

作業方針に基づいて実装を行う。通常の開発フローと同様に、コードの探索・編集・テスト実行を行う。

### 6. 作業完了後の案内

作業が完了したら、以下のように smart-commit の使用を提案する:

```
作業が完了しました。コミットする場合は `/smart-commit` を実行してください。
```

- **勝手にコミットしない** — ユーザーが明示的に `/smart-commit` を実行するか、コミットを指示するまで待つ
- ユーザーが追加の修正を希望する場合はそのまま作業を続ける

## 注意事項

- Issue の内容が不明確な場合は、実装に入る前にユーザーに確認する
- `--no-verify` は使わない
- force push はしない
- リモートへの push はしない（ユーザーが明示的に指示した場合のみ）
- 作業スコープは Issue に記載された内容に限定する（スコープ外の改善は行わない）
