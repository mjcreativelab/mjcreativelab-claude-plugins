# Phase 2: 公式推奨差分 + コンテキスト管理 + 静的チェック手順

## 前提確認（読み込み時に必ず実行 — 失敗時は中止）

Phase 2 を開始する前に、以下の 2 点を **両方** 確認する。いずれかを満たさない場合、Phase 2 を実行せず **本スキル全体を中止する**（Step 2 に戻るのではなく、ユーザーに中止を報告して終了する）。

1. **Phase 1 レポートファイルの存在**: `/tmp/skill-improver-phase1-{スキル名}.md` を Read する。ファイルが存在しない場合、Phase 1 が未実行である
2. **skill-creator 呼び出し履歴**: 会話コンテキスト内に `Skill` ツールで `skill-creator:skill-creator` を呼び出し、正常に完了した記録があること

**注意**: SKILL.md を読んで手動で Phase 2 の手順を適用する使い方は想定外である。Phase 2 の検証は Phase 1 の出力（改善後のスキル + レポート）を前提としており、Phase 1 なしでは誤った結果になる。

## Step 3a: 改善後のスキルを読み込む

Phase 1 でコンテキスト圧縮が発生している可能性があるため、すべてのファイルを読み直す。

1. `{スキルパス}` 内のファイルを `Glob` で列挙する（パターン: `{スキルパス}/**/*`。パスが不明な場合は `AskUserQuestion` で確認）
2. SKILL.md を Read する（Phase 1 のキャッシュに頼らない — コンテキスト圧縮で失われている可能性がある）
3. frontmatter `name` から `{スキル名}` を確定する
4. 補助ファイル（`references/`、`assets/` 等）を一覧として把握する（この時点では Read しない）

## Step 3b: 公式ベストプラクティス調査

最新の Claude Code 公式推奨と対象スキルの構成を照合する。Phase 1 が機能的な問題を検出するのに対し、本ステップは「古いパターン残留・新機能の取りこぼし」を検出する。

1. `WebSearch` で以下のクエリを順に実行し、最新の SKILL.md 規約・推奨パターンを収集する:
   - `Claude Code SKILL.md best practices`
   - `Claude Code skill frontmatter`
   - `Claude Code skill conventions site:docs.anthropic.com`
2. 検索結果から公式ドキュメント・リリースノート・Anthropic 公式リポジトリの URL を特定し、`WebFetch` で本文を取得する
3. 取得できない / 結果が薄い場合は、本ステップの結果を「未確認」とし、Step 3c 以降の機械的チェックに集中する

**調査観点**:

- **新しい frontmatter フィールド**: `description` / `argument-hint` / `allowed-tools` 以外の公式サポートフィールド（`context`, `agent`, `model`, `disable-model-invocation`, `user-invocable` 等）が活用できるか
- **動的コンテキスト注入の活用**: `` !`command` `` 構文や `${CLAUDE_SKILL_DIR}` / `${CLAUDE_SESSION_ID}` などの最新変数が使えるか
- **`disable-model-invocation` の設定漏れ**: 副作用のあるスキルに付与されているか
- **コンテキスト効率の新パターン**: `references/` への切り出し方針、`assets/` の使い分けが最新ガイドに沿っているか
- **`description` の発火精度**: 公式が推奨するキーワード列挙パターン・否定形（"〜とは別物"）の活用度
- **Skill ツールの呼び出し規約**: スキルから別スキルを呼ぶ際の最新規約（プラグイン名前空間 `plugin:skill` の指定）

## Step 3c: コンテキスト管理チェック

Phase 1 レポート（`/tmp/skill-improver-phase1-{スキル名}.md`）の知見を踏まえて検証する。Phase 1 がエラーで中断した場合は知見なしでそのまま進む。

SKILL.md の構造を [context-checklist.md](context-checklist.md) の観点で検証する。チェックリストを Read し、「該当条件」に合致する項目のみ検証する。リファレンス型スキル（規約・パターン定義のみ）で該当項目がなければ、その旨を報告して Step 3d に進む。

## Step 3d: 静的チェック

SKILL.md およびスキルディレクトリ内のファイルに対して以下を検証する。

| チェック項目 | 方法 |
|---|---|
| TODO / FIXME / TBD 残留 | スキルディレクトリ全体を `Grep` で検索（パターン: `TODO\|FIXME\|TBD\|後で書く\|WIP`）。`output_mode: "content"` で検索し、マッチ行がチェック手順の説明文自体（例: 本 phase2-steps.md 内のこの表）なら除外する |
| テンプレート未カスタマイズ | `<placeholder>`、`YOUR_`、`REPLACE_ME`、`example.com` 等のプレースホルダーパターンを Grep で検索。同上、チェック手順の説明文自体に含まれるマッチは除外する |
| 参照ファイルの存在確認 | SKILL.md 内のリンク（`[text](path)` 形式）の参照先がすべて存在するか Glob で確認 |
| シェルスクリプト構文 | `assets/` や `scripts/` 内の `.sh` ファイルがあれば `bash -n` で構文チェック。Bash ツールが利用不可の場合はレポートに「未実施（要手動確認）」と記載する |

## Step 3e: レポート出力（A/B/C カテゴリ分類）

Step 3b〜3d で発見した問題を以下の 3 カテゴリに分類してレポートする。

### A. 即座に適用できる改善

- 既存の構成を変更するだけで効果がある項目（例: TODO 残留の削除、リンク切れの修正、`disable-model-invocation` の付与漏れ、frontmatter のキーワード追加）
- リスクが低く、ロールバックも容易

### B. 設計検討が必要な改善

- 構造変更を伴う項目（例: SKILL.md の `references/` 分割、Phase 設計の見直し、API レスポンス絞り込みの導入、サブエージェント委譲化）
- 影響範囲が広く、設計判断が必要

### C. 情報提供のみ

- 現在は不要だが、将来的に有用な可能性がある項目
- 公式の新機能で、対象スキルでは現状必須ではないもの

問題が 10 件を超える場合は重要度「高」「中」のみ詳細化し、「低」はカウントのみ記載する。問題がなければ「Phase 2: 問題なし」と報告して終了する。

レポートは `/tmp/skill-improver-report-{スキル名}.md` にも書き出す（テンプレート: [../assets/report-template.md](../assets/report-template.md)）。書き出し後は会話にはレポートの要約（A カテゴリの件数とハイライトのみ）を残し、詳細は `/tmp` ファイルを参照する。Step 3f の修正中にコンテキスト圧縮でレポートが失われた場合はこのファイルを再 Read する。

重要度の基準:

- **高**: 実行時エラーまたはコンテキスト溢れに直結する（参照ファイル不在、再 Read 未定義で必須データ消失）
- **中**: 実行はできるがコンテキスト効率が悪い（一括読み込み、レスポンス未絞り込み）
- **低**: 将来の保守性に影響する（肥大化リスク、TODO 残留）

## Step 3f: 修正

問題が見つかった場合、`AskUserQuestion` で次のアクションを確認する:

- **A をすべて修正**（推奨）— A カテゴリの即適用項目だけ Edit で修正する
- **A と B の高だけ修正** — 設計検討が必要な B のうち重要度「高」の項目も含めて修正する
- **修正しない** — レポートのみで終了する

ユーザーの選択に従い、修正した場合は修正内容を会話に簡潔に報告する（詳細はレポートファイル参照）。
