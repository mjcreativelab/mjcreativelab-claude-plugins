# code-reviewer

実装されたコードを **仕様整合・設計適合・可読性** の観点でレビューするスキル。PR / 変更差分のレビュー、コミット前のセルフレビューに使う。対象が PR なら確認ゲート経由で GitHub の PR Review として投稿する。

## どんなケースで使うか

- PR を出す前に **セルフレビュー** したい
- 他者の PR を **Claude の観点でレビュー** したい
- 特定パスの差分だけレビューしたい
- ブランチ全体の commit 範囲をまとめてレビューしたい
- Codex と併用してクロスチェックしたい（重要変更時）

使わない場面:

- **認証・認可・決済・データスキーマ・外部 API** 等の重要変更の最終ゲート → `/code-reviewer-adversarial`
- 設計レベルの検討 → `/software-architect`
- 設計レベルのセキュリティ監査 → `/security-auditor`
- コンテキスト隔離での独立監査（会話バイアスを排除したい）→ `code-reviewer` subagent

## 使い方

```
/code-reviewer
/code-reviewer <PR番号>
/code-reviewer <branch>
/code-reviewer <ref>..<ref>
/code-reviewer <path>
```

### 例

```
/code-reviewer                                   # 現在の未コミット変更 + 現在ブランチ vs main
/code-reviewer 42                                # PR #42
/code-reviewer main..HEAD                        # 現在ブランチの全コミット
/code-reviewer packages/mjc-git-workflow-tools/  # 特定パス
```

## 引数

| 判定                           | 解釈                                          | 書き出しモード |
| ------------------------------ | --------------------------------------------- | -------------- |
| 省略                           | 未コミット変更 + 現在ブランチ vs `main`       | 現在ブランチに open PR があれば有効 |
| 数字のみ                       | GitHub PR 番号                                | 有効           |
| `<ref>..<ref>`                 | commit 範囲の diff                            | 無効           |
| ローカル / `origin/` のブランチ名 | そのブランチ vs `main`                        | 当該ブランチに open PR があれば有効 |
| ファイル / ディレクトリパス    | そのパス配下の変更のみ                        | 無効           |

## 観点

- **仕様整合** — 要件・設計ドキュメントとの一致
- **設計適合** — 既存アーキテクチャ・コンベンションとの整合
- **可読性** — 命名・構造・コメント（最終仕様のみ。履歴コメントは NG）
- **テスト** — カバレッジの妥当性、境界条件
- **オーバーエンジニアリング** — 過剰な抽象化・未使用の拡張性
- **横断影響** — skills・設定・他ドメインへの影響漏れ

## 出力フォーマット

指摘は以下の 5 区分で整理する（該当なしの区分も「なし」と明記）:

- 🚫 **ブロッカー**（マージ不可の問題）
- ⚠️ **推奨**（改善すべきだがマージは可能）
- 💬 **nit**（好みの範囲・次回対応可）
- ✅ **Good**（良い実装・判断）
- 🔄 **横断影響**（skills・設定・関連ドメインへの影響漏れ）

認証・認可 / 決済 / スキーマ / 外部 API のいずれかを含む場合、`### Codex クロスチェック推奨` セクションを追加する。

## PR 書き出しモード

対象が GitHub PR と判別できた場合、`AskUserQuestion` の確認ゲートを経て PR Review として投稿する。

- 投稿ツール: `mcp__plugin_github_github__pull_request_review_write`（`event: "COMMENT"`）
- 識別マーカー `<!-- claude-code-review:code-reviewer -->` を先頭に付与
- approve / request_changes は **人間レビュアーの判断に残す**（Claude は投稿しない）

失敗時（MCP 未接続 / 権限不足 / 複数 PR 等）は書き出しモードを無効化し、ローカル出力のみ残す。**PR 投稿が失敗してもレビュー処理自体は止めない**。

## 関連スキル

- `/code-reviewer-adversarial` — Breaker × Judge の敵対的レビュー（重要変更の最終ゲート）
- `/software-architect` — 実装前の設計
- `/security-auditor` — 設計レベルのセキュリティ監査
- `smart-review`（`mjc-git-workflow-tools`）— ローカルブランチのセルフレビュー（PR 投稿機能なし）

## 元になったスキル

特定のスキルをベースにしたものではなく、本リポジトリ独自の設計。`CLAUDE.md` で定義している「AI Agent Role Assignment」の **Claude Code = レビュー責任者（仕様整合・設計適合・可読性）** の役割を固定フォーマットで出力するために作成した。

重要変更で Codex クロスチェックを併用する前提の運用に最適化してある（`### Codex クロスチェック推奨` セクションが出たら手動で Codex レビューを回す、または `/code-reviewer-adversarial` に切り替える）。

## 推奨: 新規セッションで実行する

PR / 差分のレビューは **作業セッションとは別の新規セッションで実行する** ことを推奨する。

- 作業セッションにはコード探索・編集・試行錯誤の履歴が蓄積されており、レビュー時にそれらがバイアスになる
- 新規セッションなら、必要な diff と仕様ドキュメントだけを読み込むためトークン効率も良い
- **コミット後 / PR 作成後に新規セッションで `/code-reviewer` を実行するのがベスト**
