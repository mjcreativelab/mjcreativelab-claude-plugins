# software-architect

要件やスペックから「あるべきソフトウェア設計」を言語化するスキル。実装コードは書かず、設計判断の土台となるドキュメントを出力する。

## どんなケースで使うか

- 新機能の **設計着手時** に、データモデル / API / 責務分割を言語化したい
- 既存機能を **再設計** する前に、設計意図とトレードオフを整理したい
- 仕様の **解像度を上げる** ために、曖昧な要件を具体的な設計に落としたい
- 実装タスクを **Codex（閉じたタスク）** と **Cursor + 開発者（横断タスク）** に振り分けたい

使わない場面:

- 画面デザイン（UI / UX）の検討 — 本スキルの対象外
- 実装コードの記述 — Codex または Cursor が主担当
- 既存コードベースでの詳細な実装可能性検証 — 後段の Codex レビューで扱う

## 使い方

```
/software-architect
```

または「設計して」「アーキテクチャ検討」「この要件の設計案」と伝える。

会話コンテキスト（要件・議論内容・制約）を前提に動作するため、**事前に要件を会話内で共有しておく** のが望ましい。曖昧な点はスキル側から `AskUserQuestion` で確認する。

## 出力フォーマット

```markdown
## 設計サマリ
- 目的 / スコープ / 非スコープ

## データモデル / API / 主要コンポーネントの責務

## 主要トレードオフと選択理由
| 選択肢 | メリット | デメリット | 採否 |

## 実装タスク振り分け
- 閉じたタスク（→ Codex）
- 横断タスク（→ Cursor + 開発者）

## 想定リスクと未解決事項
```

## 手順

1. 要件の確認（`AskUserQuestion` で解像度を上げる）
2. 既存設計との整合確認（ADR・`CLAUDE.md`・関連コード）
3. 設計案の作成（データフロー / 責務分割 / インターフェース）
4. 代替案の比較（主要トレードオフと選択理由）
5. タスク振り分け判断（Codex 向き / Cursor 向き）
6. 出力

## 原則

- 抽象度を上げすぎない。具体的な判断が下せるレベルまで落とす
- 不確実な点は明示する（後段の Codex 実装観点レビューで埋める前提）
- 会話コンテキストを最大限活用する

## 関連スキル

- `/code-reviewer` — 実装後のレビュー。仕様整合・設計適合を検証する
- `/code-reviewer-adversarial` — 重要変更の敵対的レビュー（認証・決済・スキーマ・外部 API）
- `/security-auditor` — 設計レベルのセキュリティ監査（STRIDE・信頼境界）

`/software-architect` → 実装 → `/code-reviewer` → `/security-auditor` の流れで組み合わせる想定。

## 元になったスキル

特定のスキルをベースにしたものではなく、`CLAUDE.md` で定義している「AI Agent Role Assignment」（Claude Code = 設計責任者、Codex = 閉じた実装、Cursor = 横断タスク）に沿って、設計フェーズの成果物を固定フォーマットで出せるように設計した独自スキル。

## 推奨: 会話セッションの中で使う

このスキルは **会話コンテキストに強く依存する** ため、新規セッションではなく、要件議論を行ったセッション内で実行するのが望ましい。

- ユーザーとの議論内容・制約・スコープ判断を直接読み取って設計に反映する
- コンテキスト隔離での独立監査が必要な場合はプロジェクトの `code-reviewer` / `security-auditor` subagent を別途使う（`software-architect` は常に会話コンテキストが必要なため subagent 版は用意していない）
