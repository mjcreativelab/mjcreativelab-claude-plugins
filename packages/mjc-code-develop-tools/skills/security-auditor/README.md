# security-auditor

脅威モデル・認可・データフロー・設計リスクの観点から **設計レベルのセキュリティ監査** を行うスキル。STRIDE で脅威を洗い出し、影響 / 発生可能性 / 優先度を 3 段階で評価してリスクを可視化する。

## どんなケースで使うか

- 新機能の **設計時** に、脅威モデルと信頼境界を言語化したい
- **外部接点**（API・Webhook・ファイル取込み等）を変更するときにリスクを整理したい
- **認証・認可・データ扱い**（PII・秘密情報）に触れる変更の前にリスクを洗い出したい
- 設計段階 / 実装段階 / 運用段階で取るべき緩和策を整理したい

使わない場面:

- 実装レベルの脆弱性詳細スキャン（インジェクション・秘密情報のハードコード・依存脆弱性等）→ Codex に委譲
- PR / 差分のコードレビュー → `/code-reviewer`
- セキュリティツールの設定手順 → 別タスク
- コンテキスト隔離での独立監査 → `security-auditor` subagent

## 使い方

```
/security-auditor
```

または「セキュリティ監査」「脅威モデル作って」「このフローの信頼境界を整理」と伝える。

会話コンテキスト（対象コンポーネント・データの種類・変更範囲）を前提に動作するため、**事前に監査対象を会話内で共有** しておくのが望ましい。

## 手順

1. **対象の特定** — 会話コンテキストから監査対象・変更範囲を特定
2. **信頼境界の整理** — データフロー上の境界を「起点 → 終点」で特定
3. **脅威分析** — STRIDE で脅威を洗い出す（該当する分類のみ記述、全 6 分類を無理に埋めない）
4. **リスク評価** — 影響 / 発生可能性 / 優先度を **高 / 中 / 低** の 3 段階で評価
5. **緩和策の提案** — 設計段階 / 実装段階 / 運用段階で分類
6. **補助連携の提案** — Codex（実装レベル）/ Gemini（法務・代替設計・横断盲点）に委譲する観点を明記

### 優先度マトリクス

| 影響＼発生可能性 | 高   | 中   | 低   |
| ----------------- | ---- | ---- | ---- |
| **高**            | 高   | 高   | 中   |
| **中**            | 高   | 中   | 低   |
| **低**            | 中   | 低   | 低   |

## 観点

- **脅威モデル** — STRIDE / データフロー上の信頼境界
- **認可** — 権限設計・多層防御・最小権限原則
- **データフロー** — PII・秘密情報の経路と保管・ログ出力
- **設計リスク** — TOCTOU、Confused Deputy などの既知の設計上の落とし穴

## 出力フォーマット

```markdown
## 監査対象

## 信頼境界

## 脅威一覧
| # | STRIDE 分類 | 対象 | 脅威 | 影響 | 発生可能性 | 優先度 |

## 緩和策
### 設計段階で対処
### 実装段階で対処（→ Codex 補助）
### 運用段階で対処

## 補助連携
- Codex へ:
- Gemini へ:
```

## 原則

- 単一モデルでの判断に固執しない。別系統の観点を常に意識する
- 具体例なき一般論（「認可を強化する」等）を避け、対象コンポーネント・データを特定する
- 会話コンテキスト（設計意図・ユーザーとの議論）を踏まえて監査する
- 実装レベルの脆弱性は Codex に委譲し、設計レベルのリスク可視化に集中する

## 関連スキル

- `/software-architect` — 設計時に本スキルと組み合わせて使う
- `/code-reviewer` — 実装後のコードレビュー
- `/code-reviewer-adversarial` — 重要変更の敵対的レビュー（Breaker の Security persona が類似観点をカバー）
- `codex:rescue` — 実装レベルのセキュリティスキャンを委譲する先

## 元になったスキル

特定のスキルをベースにしたものではなく、本リポジトリ独自の設計。業界標準のフレームワークを組み合わせて固定フォーマットに落とし込んでいる:

- **STRIDE** — Microsoft の脅威モデリングフレームワーク（Spoofing / Tampering / Repudiation / Information disclosure / Denial of service / Elevation of privilege）
- **信頼境界（Trust Boundary）** — データフロー図でのセキュリティ境界の可視化
- **優先度マトリクス** — 影響 × 発生可能性の 2 軸でリスクを 3 段階に落とす古典的な手法
- `CLAUDE.md` の「AI Agent Role Assignment」で定義している **Claude = 設計レベル / Codex = 実装レベル / Gemini = 盲点補完** の役割分担を反映

## 推奨: 会話セッションの中で使う

このスキルは **会話コンテキストに強く依存する** ため、新規セッションではなく、設計議論を行ったセッション内で実行するのが望ましい。

- ユーザーとの議論内容・設計意図・データの種類を直接読み取って監査に反映する
- コンテキスト隔離での独立監査（会話バイアス排除）が必要な場合は、プロジェクトの `security-auditor` subagent を別途使う
