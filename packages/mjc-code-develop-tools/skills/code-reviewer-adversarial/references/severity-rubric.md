# 重大度・確信度ルブリック

Breaker と Judge が共通で参照する基準。ブレを抑えるため、Breaker は「観察可能な事実」から重大度を推定し、Judge は「修正コストに見合うか」まで踏み込んで再判定する。

## 重大度（Severity）

| レベル | 定義 | 例 |
|---|---|---|
| **Critical** | データ破壊・認証突破・秘密情報漏洩・本番サービス停止に直結 | 認可チェック欠落で他テナントデータ閲覧可、SQL Injection、PII ログ出力、マイグレーションでデータ欠損 |
| **High** | 主要機能の機能不全・明確な仕様違反・回避不能なパフォーマンス崩壊 | 正常系で再現するバグ、N+1 による本番レベル遅延、後方互換性を破る API 変更 |
| **Medium** | エッジケースの不具合・限定条件下での劣化・仕様曖昧領域の不整合 | 特定ロケールで fail、大量入力時のみ遅延、エラー分類の抜け |
| **Low** | 稀な入力・復旧容易・ユーザー影響が軽微 | ログメッセージの誤り、非本質的な境界値 |

## 確信度（Confidence）

| レベル | 定義 |
|---|---|
| **High** | failing テストで反証済み、または仕様書と diff の直接比較で矛盾が明確 |
| **Medium** | コード読解と既知のパターンから論理的に導けるが、実行での反証は未 |
| **Low** | 推測・感想レベル。`[UNVERIFIED]` タグ必須。Judge によりノイズ判定される想定 |

## Judge 側の追加判定

Judge は上記に加えて **修正コスト** を付ける:

| コスト | 目安 |
|---|---|
| **S** | 数行の修正＋局所テストで収まる |
| **M** | 数十行、既存テスト改修、周辺ファイルにも波及 |
| **L** | 設計変更・データ移行・複数サービス協調が必要 |

## 真の欠陥 / 低優先度 / ノイズの境界

Judge が分類に迷う基準を固定する:

- **真の欠陥**: 重大度 Critical/High → 原則ここ。Medium でも確信度 High なら真の欠陥
- **仕様未定**: 指摘の前提が仕様書・ADR・会話コンテキストに無い。修正前に仕様合意が必要なもの
- **低優先度**: 重大度 Low、または Medium で修正コスト L のもの。メンテナンスチケットに逃がす判断
- **ノイズ**: 確信度 Low かつ反例なし、または「感想・一般論・可読性」レベルの指摘

確信度 High なのに「ノイズ」と判定する場合、Judge は「なぜ反例があっても実害がないか」を明示的に書かなければならない（誤った切り捨ての検証可能性を担保するため）。
