# Judge Prompt Template

Phase 2 で `codex:rescue` スキル経由で Codex に渡すプロンプトのテンプレート。`codex:rescue` を呼び出す際に、下記ブロックを task として渡す。

---

## codex:rescue への依頼文テンプレート

```
あなたは adversarial code review の最終 Judge として振る舞う。
Claude による Breaker の指摘を裁定し、「真の欠陥」のみを抽出するのが目的である。

## 入力

### 1. 変更 diff
<Phase 0 で取得した git diff を貼る>

### 2. 仕様・設計意図
<関連 Issue / ADR / 会話コンテキストから抽出した要約>

### 3. Breaker 報告書（Claude Opus 4.7）
<7フィールド形式の指摘一覧を貼る>

### 4. 生成された反例テストと実行結果
<反例テストコードと、実行モードであれば fail/pass の結果>

## あなたの仕事

各指摘を以下の4カテゴリのいずれかに分類せよ:

- **真の欠陥**: 仕様違反・セキュリティリスク・性能問題として妥当で、修正価値あり
- **仕様未定**: 仕様が曖昧で、Breaker が勝手な前提を置いている
- **低優先度**: 妥当だが重大度が低く、修正コストに見合わない
- **ノイズ**: 反証不能・誤解・的外れ

## 制約

- 分類理由を1〜3文で必ず添える
- 修正コスト（S/M/L）を見積もる（S: 数行 / M: 数十行+テスト / L: 設計変更）
- Breaker が failing テストで反証した指摘は原則「真の欠陥」または「仕様未定」に振り分ける（ノイズ判定には特別な理由を要する）
- 共犯化を避けるため、Claude が書いた文面の言い回しに引きずられず、コード本体と仕様を一次情報として裁定すること
- 「良いコード」「可読性」系の指摘は評価対象外。入ってきたら「ノイズ」に振って理由に明記する

## 出力フォーマット

```markdown
## Judge 裁定結果

### 真の欠陥
#### 指摘 N: <Breaker 指摘のタイトル>
- **Judge 分類**: 真の欠陥
- **分類理由**: <1-3文>
- **修正コスト**: S|M|L
- **Breaker 7フィールドをそのまま再掲**

### 仕様未定
（同形式）

### 低優先度
- 指摘 N: <タイトル> — <分類理由>（1行）

### ノイズ
- 指摘 N: <タイトル> — <除外理由>（1行）

### 着手順序の推奨
重大度 × 修正コストから、真の欠陥の修正順序を提案
```
```

## 呼び出し時の注意

- `codex:rescue` にはこのテンプレート全体を task として渡し、裁定結果を Markdown で返させる
- Codex 側で追加のコード変更は行わせない（レビュー裁定のみ）
- Judge の結果は Phase 3 でそのまま最終出力に埋め込む。Claude 側で再解釈や書き換えはしない（共犯化回避）

## codex:rescue 呼び出し不能時の扱い

`codex:rescue` が呼び出せない環境（Codex CLI 未設定・ネットワーク断・サブエージェント内で Skill 呼び出し不可など）では、Phase 2 で停止する。

- **Phase 3 を出力しない**（Claude 単独で模擬裁定すると共犯化してしまう）
- ユーザーには次の 2 点を提示して終了する:
  1. Breaker 報告書（Phase 1 の成果物）
  2. このテンプレートを埋めた Judge 呼び出し用 task 文字列
- ユーザーが別セッションで `codex:rescue` を呼ぶか、環境を整えて再実行できるよう案内する
